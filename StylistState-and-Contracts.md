# StylistState and Contracts
Relevant source files

- [.env.example](https://github.com/domap/digital-stylist/blob/c8fd6fe5/.env.example)

The `digital_stylist` worker uses a structured state management pattern to coordinate data flow between multiple specialized agents in a LangGraph pipeline. This page documents the central state schema, the contract for agent execution contexts, and the functional patterns used to update the graph state.

## StylistState Schema

The `StylistState` is a `TypedDict` that serves as the shared bus for the entire multi-agent graph. It tracks the conversation history, classification metadata, and the specific outputs generated by domain agents (e.g., product recommendations, appointment IDs, or email drafts).

### Field Ownership and Reducers

Most fields in `StylistState` use a "last-writer-wins" strategy, where an agent's return value overwrites the previous value. The `messages` field is a notable exception, utilizing the `add_messages` reducer to append new interactions to the conversation history.

### Literal Types for Routing

To ensure type safety during graph transitions, the state utilizes specific `Literal` types for intent classification and node navigation.

- `IntentLiteral`: Categorizes the user's request into domains such as `styling`, `catalog_search`, `appointment`, `email_followup`, or `general_support`[digital_stylist/contracts/state.py#10-18](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L10-L18)
- `NextNodeLiteral`: Explicitly directs the graph to the next logical agent or the `__end__` state [digital_stylist/contracts/state.py#21-24](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L21-L24)

## AgentRunContext

The `AgentRunContext` is a Pydantic model passed to agents at runtime. It encapsulates external dependencies and configuration that do not belong in the persistent graph state, such as LLM providers and the Model Context Protocol (MCP) runtime.

## State Update Patterns

The system follows a Partial-Dict Return Pattern. Agents are not required to return the full `StylistState`. Instead, they return a dictionary containing only the fields they wish to update. LangGraph merges these partial dictionaries into the global state.

### Message Reduction

The `messages` field is annotated with `add_messages`. When an agent returns `{"messages": [new_msg]}`, the `add_messages` function handles the logic of appending the new message to the existing list, rather than replacing the list [digital_stylist/contracts/state.py#31](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L31-L31)

## Implementation Details

### Intent and Urgency

The `IntentAgent` is responsible for two critical fields that dictate graph behavior:

1. `intent`: A `IntentLiteral` used by conditional edges in `graph.py` to route the execution path [digital_stylist/contracts/state.py#38](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L38-L38)
2. `urgency_score`: An integer (1-5) used to prioritize requests in the associate-facing Clienteling app [digital_stylist/contracts/state.py#44](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L44-L44)

### Metadata and Context

The `customer_id` and `context_metadata` fields allow the graph to maintain tenant and user awareness across turns. `context_metadata` is a flexible `Dict[str, Any]` used for pass-through information from the Orchestration Gateway [digital_stylist/contracts/state.py#56](https://github.com/domap/digital-stylist/blob/c8fd6fe5/digital_stylist/contracts/state.py#L56-L56)
